The dominant forward-pass CUDA kernel was a CUTLASS SGEMM matrix multiplication.

\subq{c} What non-matmul kernels account for non-trivial CUDA runtime in the forward pass?
\deliverable{A 1--2 sentence response.}
\begin{answer}
Besides matrix multiplications, non-trivial forward-pass CUDA time came from elementwise kernels such as division, \texttt{where}, add, multiply, exponentiation, sigmoid, copy/concat kernels, and reduction kernels used for max, sum, and normalization.
For the XL model at context length $1024$, matrix multiplications accounted for $86.8\%$ of GPU kernel time, leaving about $13.2\%$ for these non-matmul kernels.
\end{answer}

\subq{d} Profile a complete training step (forward, loss, backward, AdamW step). How does the matmul fraction change vs.\ inference? What about other kernels?
\deliverable{A 1--2 sentence response.}
\begin{answer}
For the small model at context length $1024$, the matmul fraction dropped from $69.6\%$ in forward-only to $64.7\%$ in the full forward+backward+optimizer step.
For the XL model, it dropped from $86.8\%$ to $82.3\%$. This happens because backward and AdamW introduce many additional pointwise/vectorized kernels, so non-matmul work becomes a larger fraction even though GEMMs still dominate.
\end{answer}

\subq{e} Compare softmax runtime to attention matmul runtime in the forward pass. How does the runtime ratio compare to the FLOP ratio?
\deliverable{A 1--2 sentence response.}
\begin{answer}
The softmax operation did not appear as one fused softmax kernel; it was decomposed into max reductions, exponentials, sum reductions, and division.
For the XL model at context length $1024$, these softmax-like kernels were roughly $4$--$5\%$ of forward GPU kernel time, while matmuls were $86.8\%$. Thus matmul runtime was about an order of magnitude larger, much less extreme than the FLOP gap, because softmax is memory-, reduction-, and special-function-heavy and runs less efficiently than Tensor Core GEMMs.
\end{answer}

\subq{f} Using Nsight memory profiling + NVTX labels, how much memory is saved for backward by a single TransformerBlock? List the 5 largest contributing operations and their \% of total. Then estimate gradient tensor memory for a TransformerBlock from the backward-pass deltas.

\begin{answer}
\begin{center}
  \includegraphics[width=\linewidth]{nsys.png}
  \small Nsight Systems memory profiling with NVTX labels.
\end{center}

For the xl model, I measured one TransformerBlock using Nsight Systems with CUDA memory
profiling enabled and PyTorch NVTX labels. Nsight reported total CUDA memory around 19 GiB,
while \texttt{torch.cuda.memory\_allocated()} reported a little over 13 GiB. I used the
PyTorch allocated-memory deltas for saved residual tensors because \texttt{memory\_allocated()}
tracks live PyTorch tensor memory, whereas Nsight also includes allocator caching,
reserved blocks, CUDA runtime memory, and temporary library workspaces. The total saved
forward memory for one TransformerBlock was about 180 MiB. The five largest contributors were
\texttt{ffn.w3 aten::linear} at 20.00 MiB (11.09\%),
\texttt{ffn.w1 aten::linear} at 20.00 MiB (11.09\%),
\texttt{ffn.silu aten::silu} at 20.00 MiB (11.09\%),
\texttt{ffn.gate aten::mul} at 20.00 MiB (11.09\%), and
\texttt{attn.softmax aten::softmax} at 16.19 MiB (8.97\%).

During the backward pass, the saved forward tensors are freed while gradient tensors are
allocated at the same time. In the Nsight trace, the memory usage decreases by about
180 MiB across a single \texttt{TransformerBlock} backward range, which approximately
matches the amount of memory saved during the forward pass. This indicates that the memory
freed from activation tensors largely cancels out the memory allocated for gradients, due
to allocator reuse and overlapping lifetimes. As a result, the net change in memory usage
does not directly reflect the gradient tensor size.

However, we still expect the gradient memory to be on the order of the parameter size of
the block, since each parameter produces one gradient tensor of the same shape. This is
much smaller than the activation memory saved during the forward pass, which scales with
batch size and sequence length. The observed behavior therefore matches expectations:
activation memory dominates in the forward pass, while gradient memory is comparatively
smaller and masked by memory reuse during backward.
\end{answer}

\problem{gradient\_checkpointing}{4}

\subq{a} For a Transformer of $N$ identical blocks, what checkpointing strategy minimizes peak activation memory (ignoring compute)? Give asymptotic peak memory and compute as a function of $N$, plus a short code sketch.
\deliverable{A 3--5 sentence description plus a short code sketch.}
\begin{answer}
  If we only care about memory and can sacrifice compute as much as we want, we can use a fully nested (recursive) checkpointing strategy: for block $i$, checkpoint the entire prefix $0,\dots,i-1$, then execute block $i$ normally. During backward, each checkpoint recomputes its prefix just-in-time, so at any moment only $O(1)$ block residuals are alive.
  Thus, the peak activation memory is
  \[
  M(N) = \Theta(1),
  \]
  which is optimal up to constants. However, the compute cost is
  \[
  C(N) = \Theta(N^2),
  \]
  since prefixes are recomputed repeatedly:
  \[
  N + (N-1) + \cdots + 1 = \Theta(N^2).
  \]

  A sample code looks like
\begin{algorithm}[H]
\caption{Recursive checkpointed prefix evaluation}
\begin{algorithmic}[1]
\Procedure{RunPrefix}{$\textit{blocks}, i, x$}
  \If{$i = 0$}
    \State \Return $\textit{blocks}[0](x)$
  \EndIf
  \State $h \gets \textsc{Checkpoint}(\lambda y.\ \textsc{RunPrefix}(\textit{blocks}, i-1, y),\ x)$
  \State \Return $\textit{blocks}[i](h)$
\EndProcedure
\end{algorithmic}
\end{algorithm}
\end{answer}

\subq{b} For the xl model (batch size 4, seq len 2048) with a budget of \emph{one} step of recomputation (no nesting), what is the best checkpointing block size? Validate with peak-memory profiles, comparing the next-smaller and next-larger block sizes.
\deliverable{A 3--5 sentence description plus measured peak memory.}
\begin{answer}
  For the xl config with $N=32$, the best one-step strategy is to wrap the
entire stack of 32 Transformer blocks in a single checkpoint call. In the
ideal activation-only model, checkpointing 32 blocks instead of 31 blocks saves
only one extra residual stream tensor:
\[
4 \cdot 2048 \cdot 1600 \cdot 2 \text{ bytes}
\approx 25 \text{ MiB}
\approx 0.024 \text{ GiB}.
\]
This is tiny relative to the full training-step peak, which is dominated by
persistent model parameters, gradients, optimizer state, CUDA allocator
reserved memory, and temporary library workspaces. Consistent with this, my
profiles showed essentially no measurable peak-memory change between the
largest checkpoint size and the next-smaller checkpoint size. Thus I used one
checkpoint around all 32 blocks, but the measured peak memory was effectively
unchanged because the saved activation difference was too small compared with
the optimizer-dominated baseline.
\end{answer}
